# Endpointing / Continuation Segmentation — Design (Spec 2 of 3)

- **Date:** 2026-06-08
- **Status:** Approved (brainstorming) → ready for implementation plan
- **Scope:** Stop multi-fragment questions — and interviewer follow-ups on an already-answered turn — from splitting into separate, context-poor answers (issue C). Builds on Spec 1 (Conversation Core, merged PR #15).
- **Out of scope (separate spec):** boundary-classifier accuracy / prompt / observability (issue D, Spec 3).

## 1. Problem

A single interviewer question often arrives as several VAD windows: the Silero `VadWindowAccumulator` flushes a `SpeechWindow` after ~375 ms of silence (or 7.5 s max), Whisper transcribes each window, and each becomes one `TranscriptItem`. So one spoken question is delivered to the pipeline as **several fragments**.

Today, once the first answer has fired (the turn leaves `CollectingQuestion`), a later continuation fragment is mis-handled: `RouteLabel`'s `QuestionContinued` case falls through to `HandleNewQuestion` when the turn is not `CollectingQuestion` (a stale-status workaround from before Spec 1), and `HandleClarification`'s interviewer branch sends a clarification on an already-answered turn to `HandleNewQuestion` (lines 278–282). The result: **the rest of the question splits off into a separate, context-poor card** — and the second card can answer the wrong thing.

Spec 1 made the pipeline's in-memory turn **status accurate** (the `ITurnStatusFeedback` channel) and gave each turn its **own cancellation token** (`_turnCts`). That unlocks the correct fix here: route continuation/refinement fragments back onto the same turn and regenerate it.

## 2. Goals / Non-goals

**Goals**
- A question delivered in N fragments produces **one** turn/card whose answer reflects the whole question.
- An interviewer follow-up/clarification on an already-answered turn **refines that turn**, with full context (the original question + the candidate's spoken answer + the clarification) — it does not split into a context-free new card.
- Regeneration is **coalesced** so a burst of fragments collapses into a single regeneration (no flicker, no wasted API calls).
- When a fragment genuinely **is** a new question, it still opens its own turn (no merging).

**Non-goals (deferred)**
- Boundary-classifier accuracy/prompt/observability (Spec 3). Spec 2 **trusts the classifier labels**; it does not add a time/recency guard to second-guess them.
- Pre-completion collecting (`CollectingQuestion`) and the 8 s force-complete timer — kept as-is.
- `Me` routing (Spec 1), the legacy 3-state `SegmentAccumulator` fallback, and any UI/overlay change.

## 3. Behavior contract (decisions)

1. **First answer fires fast** — unchanged. A fragment classified `QuestionComplete`/`TaskComplete` fires the preliminary answer immediately.
2. **Append + debounced regeneration** for any label meaning "this belongs to the current turn", evaluated against the now-accurate in-memory status:
   - `QuestionContinued` on a turn that is **not** `CollectingQuestion` but is non-terminal or already-answered (`Generating*`/`*Ready`);
   - `AdditionalRequirement` on an answered turn;
   - `ClarificationOfCurrentQuestion` **from the interviewer (`Other`) on an already-answered turn** (the case lines 278–282 currently mis-route to a new question).

   Each appends its text to the turn's question and schedules a **debounced regeneration** (~1 s, reset on each new fragment). A burst collapses into one regeneration; fragments spaced wider than the debounce produce a regeneration after each quiet gap. The first answer is never delayed — only the *continuation* regenerations are debounced.
3. **Continuation vs. new question = classifier label only.** `NewQuestion` opens a new turn (no merging). No time/recency window. (Residual risk — a continuation mislabeled `NewQuestion` still splits — is bounded by §3.4 and is Spec 3's domain to reduce.)
4. **Wider answer context** so a split-off card (or any answer) is still answered in context:
   - `GenerateAnswerHandler.recentQA`: last **3** answered turns (was 2).
   - `PromptBuilderService` per-answer clip: **~400** chars (was 200).

`CollectingQuestion` semantics (pre-first-answer fragment collection) and the existing immediate clarification-response refine (`AwaitingClarification`/`ClarificationReceived` → `RefinedAfterClarification`) are unchanged.

## 4. Components

### 4.1 `RegenDebouncer` (new — `Application/Sessions/`)
A small, single-purpose per-turn debounce built on `TimeProvider` (the host already registers `TimeProvider.System`).
- `void Touch(ConversationTurnId turnId, Action fireRegen)` — (re)arms a ~1 s timer for the turn; each call resets it. When the quiet window elapses, invokes `fireRegen` **once** and clears the entry.
- `void Cancel(ConversationTurnId turnId)` — drops a pending timer (turn went terminal / was regenerated by another path).
- `IDisposable` — disposes all outstanding timers.
- No domain state; just timing. Deterministically unit-testable with `Microsoft.Extensions.Time.Testing.FakeTimeProvider`.
- Debounce duration is a private constant (`DebounceMs = 1000`), in the spirit of the existing `MaxCollectionSeconds`; not user-configurable (YAGNI).

### 4.2 `TranscriptPipelineService` (changed — `Application/Sessions/`)
- Inject `TimeProvider`; own a `RegenDebouncer`.
- **`RouteLabel`**: the three "belongs-to-current-turn" labels (§3.2) → append to the turn's question (`AppendToQuestion`) and `RegenDebouncer.Touch(turnId, () => FireAndForget(regenCommand, …))`, instead of today's fall-through to `HandleNewQuestion` / immediate regen.
  - `QuestionContinued` while `CollectingQuestion` still `AddFragment` (unchanged pre-fire path).
  - `AdditionalRequirement` → append + `Touch` (was: immediate regen in `HandleAdditionalRequirement`).
- **`HandleClarification`** interviewer branch: when the turn is already answered (not `AwaitingClarification`/`ClarificationReceived`), route to append + `Touch` instead of `HandleNewQuestion` (fixes lines 278–282). The existing `AwaitingClarification`/`ClarificationReceived` → immediate `RefinedAfterClarification` regen is unchanged (user-/clarification-driven, not a fragment burst).
- **Guard:** append+regen only when the target turn is non-terminal or answered; if the only turn is terminal (`Dismissed`/`Resolved`), treat as a new question.
- **Cleanup:** `RegenDebouncer.Cancel(turnId)` when a turn reaches a terminal/ready-then-dismissed status, alongside the existing `_turnCts` disposal in `DrainStatusFeedback`; `Dispose()` cancels all.

### 4.3 `GenerateAnswerHandler` + `PromptBuilderService` (changed — `Application/Answers/`)
- `recentQA`: `TakeLast(2)` → `TakeLast(3)`.
- `PromptBuilderService` answer clip: `200` → `400` chars (one constant; update the doc comment that says "capped at 200 characters").

## 5. Data flow (a continuation fragment)

1. `Other` fragment arrives on the single-threaded consumer; status feedback drained (Spec 1) → in-memory turn status accurate.
2. Classifier labels it `QuestionContinued`/`AdditionalRequirement`/`ClarificationOfCurrentQuestion`.
3. Pipeline appends the text to the turn's question **and** `unitOfWork.SaveChangesAsync` — so the separate-scope `GenerateAnswerHandler` will read the full assembled text.
4. `RegenDebouncer.Touch(turnId, fireRegen)` — (re)arms the ~1 s timer.
5. Quiet ~1 s → timer fires `fireRegen` → `FireAndForget(regenCommand)` → the turn's `_turnCts` cancels any still-streaming prior answer; the regen reads the full question + widened context and streams.
6. `GenerateAnswerHandler` publishes `TurnStatusEvent`s; next consumer iteration applies them.

## 6. Worked example

```
Other  t=0.0  "How does GC work in .NET"            QuestionComplete  -> T1 created, answer A1 fires
Other  t=2.0  "and when would you tune it"          QuestionContinued -> append T1, Touch (fire @3.0)
Other  t=2.8  "in a high-throughput service"        QuestionContinued -> append T1, Touch reset (fire @3.8)
              (quiet 1s) @3.8  -> ONE regen A2 of T1 with full question  [no split, 1 regen not 2]
Me     t=5.0  "I'd use server GC and tune the heap" (mic) -> attached to T1 as context (Spec 1), no regen
Other  t=9.0  "Can you be specific about which       ClarificationOfCurrentQuestion (answered turn)
               settings for high-throughput?"        -> append T1 + Touch -> regen A3 with
                                                          full question + my answer + the clarification
Other  t=14   "Different topic — Kubernetes?"        NewQuestion -> new card T2 (T1 untouched)
```

If `t=9.0` were instead mislabeled `NewQuestion`, it opens a card T2 — but `recentQA` (now 3 deep) carries T1's Q+A and `recentTranscript` carries the candidate's answer, so T2 is still answered **in context**, not blindly.

## 7. Error handling / concurrency

- Append + persist happen on the consumer thread. The debounce **timer callback only builds a command from immutable IDs and calls `FireAndForget`** — it never mutates the in-memory `session` off-thread. The persisted text is what the regen reads.
- Timer fires after the turn already went terminal / was regenerated elsewhere → `FireAndForget` is harmless (per-turn CTS / terminal guard); prefer to `Cancel` on terminal transition.
- `OperationCanceledException` on a superseded same-turn regen → existing handling (`answer.Cancel`).
- A continuation arriving while the first answer is still streaming → the debounced regen cancels it (per-turn CTS) and restarts with full text; this is the one bit of visible churn, once per coalesced burst.

## 8. Testing strategy

- **Unit — `RegenDebouncerTests`** (`FakeTimeProvider`): burst of 3 `Touch` within the window → exactly **1** `fireRegen`; two `Touch` spaced > window → **2**; `Cancel` before elapse → **0**; `Dispose` cancels pending.
- **Unit — `TranscriptPipelineServiceTests`**:
  - `QuestionContinued` after `PreliminaryReady` → append to the **same** turn + debounced regen, **no new turn** (the split fix).
  - `AdditionalRequirement` on an answered turn → debounced (not immediate) regen.
  - `ClarificationOfCurrentQuestion` from `Other` on an answered turn → append+regen the same turn (not a new question).
  - `NewQuestion` → still opens a new turn (no merging).
  - Terminal-only turn → continuation becomes a new question; no regen on the terminal turn.
- **Unit — answer context**: `GenerateAnswerHandler` includes the last **3** answered Q&A; `PromptBuilderService` clips answers at **400** chars.
- **E2E (from `develop`, system audio)**: a question delivered in 2–3 chunks → one card whose final answer reflects the whole question, ≤ a couple of regenerations; a clarify-after-answer from the interviewer → the same card refines with context.
